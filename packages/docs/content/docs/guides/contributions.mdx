Most of the data in the namespace is community-contributed. A `contribution` is how a change gets proposed; the Pentaract handles moderation and merging.

Every request needs an `X-Client-Key` header (see [Quickstart: Register an API client](/docs/getting-started/quickstart#register-an-api-client)).

## The model

A contribution is three things bundled together:

- A **target**: the AT-URI of the record being changed (absent if the contribution is a brand new game).
- A **patch**: a partial object whose shape matches the target record's schema. Only the fields being changed are present.
- A **message**: prose explaining what's being changed and why.

The `contributionType` field is one of:

- `correction`: fixing existing data on an existing record.
- `addition`: adding new data to an existing record (e.g., a missing release, a new screenshot).
- `newGame`: proposing a whole new game record.

Contributions are signed records in the contributor's own repo. Nobody else can write them. Once written, the Pentaract sees them, runs whatever review it runs, and decides whether to merge them into the canonical record.

## Creating a contribution

Contributions are created via the `createContribution` procedure, not by writing a record directly. This lets the Pentaract sign the contribution before it lands in the contributor's repo. Contributions that are not signed will not be sent to the review queue.

Creating a contribution requires authentication (see [Guide: Authentication](/docs/guides/authentication)).

The Pentaract creates a `contribution` record in the user's repo with its signatures attached. You don't need to manage the record key or the signatures yourself.

## Review process

When a signed contribution is created via the Pentaract, it's added to a moderation queue. The contribution can then be reviewed by a Pentaract moderator and/or the game record's owner.

### Contribution acceptance

When a Pentaract moderator accepts a contribution, the patch will be applied automatically any time the associated record is requested from Pentaract. If a Cartridge moderator rejects the contribution, it should not be counted towards the user's contribution stats.

The owner of a game record can also review a contribution. This can be done before a Pentaract moderator ever sees the contribution. The owner's decision overrides any decision made by Pentaract moderators. For example, a moderator could approve a contribution, but the contribution can later be rejected by the record owner, resulting in the patch no longer being applied.

## Provenance

Accepted contributions are not deleted. The `contribution` record stays in the contributor's repo forever; the `contributionPatch` sits next to the target record documenting exactly what changed. This gives the namespace a full audit trail. If you want to know who added a genre, when, and why, the data's right there.

## Contribution stats and badges

The Pentaract tracks contribution statistics per user. Use `getContributionStats` to fetch a contributor's stats and earned badges:

<Tab value="TypeScript">
```ts
const url = new URL(
  '/xrpc/games.gamesgamesgamesgames.getContributionStats',
  'https://gamesgamesgamesgames.games'
)
url.searchParams.append('did', 'did:plc:abc123')

const response = await fetch(url, {
  headers: { "X-Client-Key": "API_CLIENT_KEY" }
});
const { stats, badges } = (await response.json()) as {
  stats: {
    totalContributions: number;
    corrections: number;
    additions: number;
    newGames: number;
    uniqueEntities: number;
    ownerAccepted: number;
  };
  badges: Array<{ id: string; name: string; description: string }>;
};

console.log(`${stats.totalContributions} total contributions`);
console.log(`${stats.corrections} corrections, ${stats.additions} additions, ${stats.newGames} new games`);
console.log(`${stats.uniqueEntities} unique entities edited`);
console.log(`${stats.ownerAccepted} accepted by record owners`);
```

The `stats` object breaks down contributions by type (`corrections`, `additions`, `newGames`) and tracks how many distinct records the user has contributed to (`uniqueEntities`) and how many were accepted (`ownerAccepted`).

### Badges

The `badges` array contains badges the user has earned based on their contribution history. Each badge has an `id`, `name`, and `description`. Use these to build contributor profiles or leaderboards.
